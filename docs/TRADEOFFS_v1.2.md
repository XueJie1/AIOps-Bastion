# AIOps-Bastion 设计取舍说明 (Tradeoff Rationale) — v1.2

> 配合 [`DETAILED_DESIGN_v1.2.md`](./DETAILED_DESIGN_v1.2.md) 阅读。
>
> **这份文档的目的不是复述决策,而是解释"为什么是它而不是别的",以及"什么会让我改主意"。** 每一条都可被追问到第三层。若某条只能答出"因为安全/因为简单",那它就不该出现在这里——本文只收录我能讲出真实备选、真实代价、真实翻案条件的取舍。
>
> 项目定位:**求职作品集**(非生产系统)。规模锚点:**1~3 台被纳管节点 / 单用户 / 2C4G NAS**。所有取舍都挂在这三个锚点上,脱离锚点讨论"该不该"没有意义。

---

## 0. 做取舍的总原则

1. **规模先行。** 1-3 节点 + 单用户 + 2C4G 是所有取舍的坐标系。脱离这个规模谈"够不够通用"是伪命题——这个项目不打算服务 100 台机器,也不为此道歉。
2. **优先级是:安全 > 可逆性 > 泛用性 > 性能 > 功能广度。** 这个顺序本身是一个取舍:它意味着我宁可少做几种修复动作(牺牲功能广度),也不放开 shell(守住安全);宁可把节点做成 DB 文档而非常量(可逆),也不为最优性能硬编码。
3. **可逆 > 最优。** 凡是有"未来翻案路径"的地方,我都允许当下选次优——节点是 Firebase 文档不是常量、vendor 是配置不是编译期绑定、嵌入模型是可替换函数、L3 模板是 dict。**只有可逆,才敢做窄。**
4. **明确划范围外,比塞进范围更重要。** HA、多用户 RBAC、自动路由、reboot、跨节点编排——不是没想过,是这个规模下不值得。见 §5。

## 1. 核心取舍(逐条)

每条格式:`选了什么` / `真实备选` / `代价` / `理由(在此规模下)` / `何时翻案`。

---

### 1.1 云端 LLM,而非本地 `[决策#1]`

- **选了:** 云端 LLM(Glm/DeepSeek/Claude/GPT 任一),仅"结构化摘要"出网。
- **备选:** ① 本地部署 Qwen/Llama(2C4G 能跑的尺寸);② 纯规则无 LLM,只做告警→脚本映射。
- **代价:** 运维数据(日志摘要、拓扑)出网;依赖厂商可用性;API 调用成本。
- **理由:** Agent 的核心价值是**规划 + 工具调用质量**,2C4G 能跑的本地模型在这项上明显弱,会直接拉垮排查闭环。国产模型已降到单次调查 2~3 RMB,成本非约束。原始日志绝不出网(§4.5),出网的仅是 Agent 本地提取的摘要。
- **何时翻案:** 节点含强合规数据(PII/金融)→ 私有部署或本地模型;或小尺寸本地模型工具调用能力达标时。

### 1.2 封闭动词集 + 硬编码模板,而非通用 shell `[决策#7][决策#8]`

> 这是整套安全设计的基石,也是被问最多的一条。

- **选了:** L1/L2 白名单封闭动词集 + `IDENT_RE` fullmatch;L3 三个枚举动作(restart_service / restart_container / clear_cache)+ 硬编码模板。**Agent 无 shell 拼接能力。**
- **备选:** ① Agent 生成 shell + seccomp/沙箱兜底;② LLM 生成命令 + shell AST 静态分析过滤。
- **代价:** L3 只有 3 个动作,新修复类型要改代码 + 改 schema enum + 重新部署;无法处理"未预见的修复"。
- **理由:** Agent 生成 shell 的注入面**不可收敛**——编码(Base64)、多行拆分、环境变量、`$()` 嵌套都能绕过正则与 AST。shell 语法本身在静态分析下不可靠。可枚举动作覆盖目标场景 80%+,用"灵活性"换"注入面可收敛"在这条产品边界上划算。三层防线(`IDENT_RE` → asyncssh `shlex.quote` → 远端 `rbash`)是**纵深防御**,不是任一单点——任何一层失守,另两层仍拦。
- **何时翻案:** 若运维场景需要大量不可枚举的一次性修复 → 引入"**预审脚本仓库**"(预审脚本 + hash 白名单 + 参数化),而非放开 shell。这条几乎不翻成"允许任意 shell"。

### 1.3 单进程 + asyncio 信号量,而非任务队列 `[决策#3]`

- **选了:** 单进程单事件循环;`Semaphore(4)` 限 SSH 并发;`Semaphore(3)` 限调查级并发;持久化靠 LangGraph Checkpointer。
- **备选:** ① Celery + Redis 任务队列;② 多进程执行器池;③ 跨节点编排器(如 Temporal)。
- **代价:** 单进程天花板(一个 bug 全停);无跨节点原子编排;调查持久化不靠外部队列靠 SQLite 检查点。
- **理由:** 1-3 节点 + 单用户,队列是过度工程。asyncio 单进程撑 4 路 SSH 绰绰有余。**进程少 = 故障域小 = 排障简单**,这对作品集尤其重要——我能讲清整个执行路径,而 Celery + Redis + broker 的排障深度不划算。
- **何时翻案:** 节点数 >10 或多用户并发 → 引入队列解耦"调度"与"执行",`ExecutionEngine` 抽象已隔离,替换面可控。

### 1.4 LangGraph,而非纯 LangChain / 自研状态机

- **选了:** LangGraph + 持久化 Checkpointer(SQLite)。
- **备选:** ① 纯 LangChain Agent Executor;② 自研状态机 + 手写 interrupt。
- **代价:** 框架绑定 + 学习曲线;LangGraph API 仍在演进,break change 风险。
- **理由:** HITL `interrupt()` + 持久化 Checkpointer + 步骤级 token 计量是 LangGraph 原生能力。自研这三样(尤其"挂起后跨崩溃恢复")极易出错,而崩溃恢复正是这个项目的硬需求(§6.8/§11)。状态机可视化对调试 Agent 也有实打实价值。
- **何时翻案:** LangGraph 重度 break change 或性能成瓶颈 → 抽象 `GraphRunner` 接口,自研薄状态机替换。`build_llm` 那层不绑 LangGraph,LLM 调用层可独立复用。

### 1.5 HITL 在单用户下定位为"防误触确认",而非"权限隔离" `[决策#12][§6.7]`

- **选了:** 触发者与审批者为同一账号,二次点击确认;不引入 step-up 口令 / 多用户 RBAC。
- **备选:** ① step-up 操作口令(Vault 存独立口令,L3 审批时校验);② 多用户 RBAC(触发者≠审批者)。
- **代价:** 无真正权限分离;WebUI 会话被劫持则 HITL 形同虚设。
- **理由:** 单用户是既定前提,触发者与审批者**物理上**是同一人,无法实现权限隔离——强行做多账号是伪需求。HITL 在此场景的真实价值是"**防 Agent 误判导致误修复**":Agent 把症状误诊为 nginx 挂了要 restart,人在审批弹窗看到渲染命令和"目标主机是 B 不是 A"时能拦下。它是"人审机器",不是"人审人"。我不假装它能权限隔离。
- **何时翻案:** WebUI 暴露面增大(如公网可访问、会话劫持风险上升)→ 追加 step-up 口令,方案 B 已预留。

### 1.6 Firebase,而非自托管实时 DB `[决策#5]`

> 这是全篇**立场最妥协**的一处,我把它放在明面上。

- **选了:** Firebase Firestore(实时同步)+ Auth(单账号)。
- **备选:** ① 自托管 Postgres + SSE/WebSocket;② Supabase 自托管;③ Redis pub/sub + 自研 Auth。
- **代价:** 工单/日志元数据出网到 Google;与"家庭内网 Zero Trust"立场**冲突**;依赖 Google 可用性。
- **理由:** `onSnapshot` 实时同步 + Auth + 离线缓存是开箱即用,自托管等价物的开发量不小(实时、重连、离线缓存、Auth 四件套)。单用户数据量极小,免费额度足够。
- **立场冲突的诚实承认:** 凭证(Vault)绝不出网,这是硬红线;出网到 Firebase 的仅是**非敏感运维元数据**(工单状态、日志摘要、拓扑)。但"自托管纯度"确实破了——这是我用"纯度"换"开发效率 + 离线缓存"的取舍,在单用户小数据量下我认为划算,**但这是我最愿意被挑战的一条**(见 §2 Q1)。
- **何时翻案:** 要彻底自托管 → 换 Postgres + SSE,`FirebaseWriter` 抽象层已隔离写入路径,替换面可控。

### 1.7 本地 MiniLM + BM25 混合检索,而非云嵌入/大本地模型 `[§3.6]`

- **选了:** `all-MiniLM-L6-v2`(384 维)+ BM25,RRF 融合。
- **备选:** ① 云端嵌入(OpenAI/Glm embedding);② 本地 bge-m3 / multilingual-e5-large。
- **代价:** MiniLM 对中英混杂运维日志语义弱,召回率低。
- **理由:** 2C4G 跑不动 ~2GB 大模型;云嵌入违背"文本不出网";BM25 关键词检索补足语义短板。RAG 是**辅助**不是核心,冷启动可降级为通用推理,不阻塞主链路。
- **何时翻案:** 机器升级 → 换 bge-m3,**仅需替换嵌入函数与维度,混合检索框架不变**。这是预留得最干净的翻案路径之一。

### 1.8 Token 四道闸,而非简单事后检查 `[§6.6]`

- **选了:** 事前估算 + Provider 侧 `max_tokens` 硬限 + 事后真实 usage 累计 + 滑动窗口速率收敛。
- **备选:** ① 仅事后累计;② 仅 Provider `max_tokens`。
- **代价:** 实现复杂度;跨厂商 token 估算用启发式(中 chars/1.5、英 chars/4),不精确,可能误杀(估高)。
- **理由:** 每道闸都有别的道覆盖不到的盲区——纯事后无法阻止**单次**超支(一次长输出 + 长上下文就爆);Provider `max_tokens` 只限输出不限输入;滑动窗口防的是"每步都没超、但累计速率注定超"。四道组合后才完备。
- **何时翻案:** 有精确的跨厂商分词器 → 替换启发式估算,闸的框架不变。

### 1.9 固定单一 vendor,而非按任务自动路由 `[决策#19]`

- **选了:** 部署时配置单一 vendor,不按任务复杂度自动路由。
- **备选:** 简单步用便宜模型 / 难步用贵模型,自动路由。
- **代价:** 不能成本优化;单 vendor 故障即停摆(无自动 fallback)。
- **理由:** 路由逻辑(判断某步难易)本身要一次 LLM 调用,收益不抵复杂度;国产模型已便宜到成本非约束;多 vendor 维护(兼容性、分词、限速、错误码差异)成本高。
- **何时翻案:** 若引入 Claude/GPT 做难步 + Glm 做便宜步且价差显著 → Provider 抽象层已支持多 vendor 共存(§8.3 bundle),加一层路由策略即可,不需要重写。

### 1.10 L3 排除 reboot `[决策#7]`

- **选了:** 无 reboot 模板,整机重启不在 Agent 能力范围内。
- **备选:** 纳入 reboot + 强制 HITL。
- **代价:** 无法自动重启整机;整机死机需人工介入。
- **理由:** reboot 影响面远超服务重启(所有服务、SSH 短暂失联可能触发 Bastion 自身误判)。更关键:reboot 常是"**不知道根因时的粗暴手段**",不该让 Agent 触发;而真正需要 reboot 的场景(整机卡死、内核 panic)往往 SSH 都连不上,Agent 根本够不到——这本就该走带外(IPMI/PDU),不是 SSH。
- **何时翻案:** 几乎不翻。若加,也应是物理带外而非 SSH reboot。

### 1.11 单 Bastion / 无 HA `[决策#3][§11.4]`

- **选了:** 单 Bastion,`restart: unless-stopped` + 卷持久化 + 冷备。
- **备选:** ① 主备 Bastion + 状态同步;② 跨机热备。
- **代价:** 单点故障,宕机即全停。
- **理由:** 1-3 节点规模,HA 的成本(状态同步、脑裂处理、共享存储)远高于收益。监控(Uptime Kuma)本身有**独立通知通道**兜底,Bastion 挂了人能收到通知,不会静默失败。
- **何时翻案:** Bastion 服务于生产关键链路 → 主备 + 共享卷。当前规模不值得。

### 1.12 CF Tunnel 唯一入站,而非开放端口 + 反代 `[§4.4]`

- **选了:** `cloudflared` 出站长连接,Bastion 无任何公网监听端口。
- **备选:** ① 端口转发 + Caddy/Nginx 反代 + Let's Encrypt;② Tailscale/WireGuard。
- **代价:** 依赖 Cloudflare;多一个外部依赖与策略风险。
- **理由:** 家庭内网无公网 IP;Zero Trust"拒绝一切主动入站"原则;CF Tunnel 的**出站模型**天然规避端口暴露面。Tailscale 亦可但需客户端,对 webhook 来源(Uptime Kuma)不如 CF Tunnel 友好。
- **何时翻案:** Cloudflare 政策/可用性风险不可接受 → Tailscale Funnel 或自建 frp,`cloudflared` 是独立容器,替换不碰 Bastion 主进程。

### 1.13 MCP 作为唯一运维出口,而非 Agent 直连 SSH `[§3.3]`

- **选了:** 所有运维副作用经 MCP 工具;Agent 不持有 SSH 执行器。
- **备选:** Agent 直接持有 `ExecutionEngine`。
- **代价:** 多一层 stdio 子进程边界;调试复杂;性能多一跳。
- **理由:** 权限分级(L0-L3)+ 审计 + 协议标准化 + 可被其他 Agent/客户端复用。**Agent 不持有执行器 = 即便 Agent 被诱导也绕不过 PermissionGate**——L3 校验 `approval_id` 在 MCP 侧,不在 Agent 侧。这是"不依赖 Agent 自觉"的关键。
- **何时翻案:** 几乎不翻。性能开销在 1-3 节点可忽略;可测试性靠 in-process mock 解决(§10.4)。

## 2. 最容易被挑战的 5 问(预载答辩)

这五问是我在面试场最预期被追问的,也是这份文档真正的"考点"。

### Q1 "你标榜家庭内网 Zero Trust,却把工单数据放 Firebase,不矛盾吗?"

**最尖锐的一问,我直接承认矛盾。** 信任边界分两层:凭证(Vault)绝不出网,是硬红线;运维元数据(工单/日志摘要)出网到 Firebase,是为实时同步做的工程取舍。矛盾是真的,不是修辞。彻底解法是自托管 Postgres + SSE,我留了 `FirebaseWriter` 抽象层。当前选择 = 用"自托管纯度"换"开发效率 + 离线缓存 + Auth",在单用户小数据量下我认为划算。**这是我全篇最愿意让步的一条**——如果有人说"这个取舍我不接受,换自托管",我会同意,且能给出替换工作量评估。

### Q2 "单用户下 HITL 还有意义吗?自己触发自己审批不就是走过场?"

物理上确实无法权限隔离,我没假装它能。HITL 在此场景的真正价值是"**防 Agent 误判**":Agent 把症状误诊、要 restart 错误的服务,人在审批弹窗看到**渲染后的完整命令**和**目标主机**时能拦下。它是"人审机器",不是"人审人"。会话劫持风险上升时加 step-up 口令(预留方案 B)。如果有人问"那为什么不做多用户",答:多用户对这个规模是伪需求,而且会引入认证/授权/邀请流的复杂度,得不偿失。

### Q3 "L3 只有 restart/clear_cache 三个动作,这种系统能处理多少真实故障?"

诚实说,它能**自动修复**的故障范围很窄——服务卡死、容器僵死、缓存脏了。但它能**排查**的范围宽得多(L1/L2 只读探测 + RAG 几乎不受限)。设计刻意把"排查面"做宽、"自动修复面"做窄:**排查错了顶多多花 token,修复错了要业务背锅。** 窄修复面 + 宽排查面 + 人工兜底 HITL,是这个规模下风险/收益最平衡的形态。要扩修复面,走"预审脚本仓库"而非放开 shell(见 §1.2 翻案条件)。

### Q4 "3 台机器你搞四道闸 token 预算、崩溃恢复、灾难恢复,是不是过度工程?"

**这取决于项目定位。** 若是生产自用,是过度,我会先砍到两道闸 + 无崩溃恢复上 MVP。作为作品集,这些是展示"工程完备性思维"的载体——但前提是我能讲清每道闸防的是**什么具体故障模式**,否则就是堆料:事前闸防单次超支、Provider 硬限防输出爆、事后闸防累计、滑动窗口防速率失控,四者盲区不重叠。崩溃恢复防的是"进程崩在 HITL 挂起态导致工单永久悬挂"。如果我讲不出每个机制对应的具体故障,那它确实不该存在——这一条我接受被逐项追问。

### Q5 "为什么用云端 LLM 而不是本地?运维数据出网你不担心?"

担心,所以原始日志**绝不出网**,仅 Agent 本地提取的结构化摘要出网。选云端的硬理由是 Agent 的规划 + 工具调用质量,2C4G 能跑的本地模型在这项上明显弱,而 Agent 质量是整个产品价值的核心。用"数据最小化出网"换"推理能力",在这个规模下我接受。若节点含强合规数据,这条会翻案(见 §1.1)。

## 3. 信心梯度(诚实版)

| 梯度 | 取舍 | 说明 |
| :--- | :--- | :--- |
| 🟢 最有信心 | 模板+白名单三层防线 | 安全闭环逻辑完备,纵深防御任一层失守另两层仍拦 |
| 🟢 最有信心 | 四道闸 token 预算 | 盲区互补,每道闸对应不同超支模式 |
| 🟢 最有信心 | 单进程 asyncio 架构 | 规模匹配,故障域小,可讲清全链路 |
| 🟡 中等 | LangGraph 选型 | 原生 HITL+Checkpointer 划算,但框架绑定是真风险 |
| 🟡 中等 | CF Tunnel 唯一入站 | 出站模型契合 Zero Trust,但 Cloudflare 依赖是软肋 |
| 🔴 最没信心 | Firebase 云依赖 | 立场冲突最明显,我最愿意被说服换掉的一条 |
| 🔴 最没信心 | MiniLM 嵌入召回率 | 已知弱,靠 BM25 兜底,实测可能不达标 |
| 🔴 最没信心 | L3 运行时不可扩展 | 真实泛用性硬伤,新动作要改代码 |

> 把最没信心的三条放在明面上,比把它们藏起来更安全。面试官问到了,我说"这三条是我自己也最不满意的,翻案路径是 X"——这比硬撑"都很好"可信得多。

## 4. 范围外(明确不做,及为什么)

| 不做的事 | 为什么不做 |
| :--- | :--- |
| HA / 主备 Bastion | 1-3 节点规模,HA 成本远高于收益;Uptime Kuma 独立通知兜底 |
| 多用户 RBAC | 单用户既定前提,多账号是伪需求,且引入认证/授权/邀请流复杂度 |
| 多 vendor 自动路由 | 路由本身要 LLM 调用,收益不抵复杂度;国产模型已足够便宜 |
| 跨节点原子编排 | 当前 L3 是 per-host single-action;若 3 台是耦合集群才需要,且应作为独立编排层引入 |
| reboot / 任意 shell | 影响面过大 + 注入面不可收敛,见 §1.2/§1.10 |
| 自定义脚本运行时白名单 | 增加预审与 hash 管理成本,当前 3 枚举动作已覆盖目标场景;预留为翻案路径 |
| mlock / seccomp 加固 | 容器能力膨胀 + 第三方库兼容风险;基础加固在单用户内网可接受 |

**这些不是遗漏,是选择。** 把它们写出来,是为了表明"我考虑过并且决定不做",而不是"没想到"。

## 5. 一句话索引(面试前 30 秒过一遍)

1. **云端 LLM**(§1.1):要推理质量,原始日志不出网,仅摘要出网。
2. **模板+白名单**(§1.2):Agent 无 shell 能力,三层防线纵深,可枚举动作换可收敛注入面。
3. **单进程 asyncio**(§1.3):规模匹配,故障域小,不引入队列。
4. **LangGraph**(§1.4):要原生 HITL+Checkpointer,自研崩溃恢复易错。
5. **HITL=防误触**(§1.5):单用户无法权限隔离,定位为人审机器,非人审人。
6. **Firebase**(§1.6):立场最妥协的一条,用纯度换效率,留替换抽象层。
7. **MiniLM+BM25**(§1.7):2C4G 跑不动大模型,BM25 补语义,框架可换。
8. **四道闸**(§1.8):每道防不同超支模式,盲区互补。
9. **固定 vendor**(§1.9):路由收益不抵复杂度,Provider 抽象在,切换=配置。
10. **排除 reboot**(§1.10):影响面过大 + 该走带外不该走 SSH。
11. **无 HA**(§1.11):规模不值得,Uptime Kuma 独立通知兜底。
12. **CF Tunnel**(§1.12):出站模型契合 Zero Trust,无公网监听。
13. **MCP 唯一出口**(§1.13):Agent 不持有执行器,绕不过 PermissionGate。
